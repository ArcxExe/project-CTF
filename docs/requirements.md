# Capture The Flag (CTF) & Academic Grading Platform - Бизнес-требования (Requirements)

Настоящий документ содержит полное описание функциональных и нефункциональных требований к веб-платформе для проведения учебных CTF-соревнований, тестирования, учета баллов за лабораторные работы и автоматического расчета итоговых оценок студентов.

---

## 1. Модуль академической структуры и пользователей

### 1.1. Сущности академической структуры
- **Поток (AcademicFlow):** Объединяет несколько академических групп. Имеет уникальное название и учебный год.
- **Академическая группа (AcademicGroup):** Привязана к одному Потоку. Имеет уникальный шифр/название.
- **Студент (Student):** Физическое лицо, проходящее обучение. Поля:
  - Уникальный ID.
  - ФИО (Фамилия, Имя, Отчество).
  - Ссылка на `AcademicGroup` (может отсутствовать для вольнослушателей).
  - Уникальный шифр студента (Student Code).
  - Связь с сущностью `User` (1-to-1, может быть пустым до привязки).
  - Статус студента:
    - `ACTIVE` (Активен).
    - `BLOCKED` (Заблокирован — закрыт доступ к платформе).
    - `OUT_OF_RATING` (Вне рейтинга — скрыт из публичного лидерборда).
    - `DISQUALIFIED` (Дисквалифицирован — обнуление результатов во всех рейтингах).

### 1.2. Процесс регистрации и сопоставления учетных записей
1. **Предварительный импорт/создание:** Преподаватель (ADMIN) может вносить студентов вручную через интерфейс или импортировать списком из файлов CSV/Excel до того, как студенты создадут аккаунты. При этом создается запись `Student`.
2. **Регистрация пользователя:** При самостоятельной регистрации студент указывает e-mail, пароль, желаемый username и свой шифр студента (Student Code). Создается сущность `User` с ролью `STUDENT`.
3. **Сопоставление (Binding):** 
  - Система автоматически пытается привязать `User` к `Student` по совпадению шифра студента.
  - Преподаватель (ADMIN) через административный интерфейс должен подтвердить, отклонить или отредактировать эту привязку. До подтверждения привязки доступ студента к соревнованиям может быть ограничен.

## 2. Модуль учета баллов за лабораторные работы

### 2.1. Ведение успеваемости
- **Баллы за лабораторные работы ($M$):** Для каждого студента хранится текущее количество набранных баллов за лабораторные работы в семестре.
- **Редактирование баллов:** ADMIN может изменять баллы вручную (через веб-интерфейс) или загружать их массово через импорт файлов CSV/Excel.
- **История изменений:** Каждое изменение баллов логируется: фиксируется дата/время изменения, автор (ADMIN), старое значение, новое значение и обязательное текстовое основание (причина изменения).

### 2.2. Расчет показателя лабораторных работ ($v_1$)
- **Фактический максимум ($M_{max}$):** Вычисляется динамически как максимальный фактически набранный балл за лабораторные работы среди выбранной совокупности студентов.
- **Настройка совокупности:** ADMIN настраивает область поиска максимума:
  - По конкретной академической группе.
  - По академическому потоку.
  - По списку выбранных групп.
  - По всем студентам в системе.
- **Формула расчета:**
  $$v_1 = \frac{M}{M_{max}}$$
  *Примечание:* Если плановый максимум (например, 100 баллов) отличается от фактического максимума (например, лучший студент набрал 120 баллов), в качестве $M_{max}$ берется именно фактический максимум (120).

## 3. Модуль тестового этапа (Квизы)

### 3.1. Структура и проведение тестов
- **Связь с соревнованием:** Перед началом основного CTF-этапа участники проходят обязательный теоретический тест (Квиз), привязанный к конкретному соревнованию.
- **Параметры теста:** ADMIN задает:
  - Временные рамки теста (starts_at, ends_at) и лимит времени на прохождение (таймер).
  - Порядок вопросов (фиксированный или случайное перемешивание).
  - Количество баллов за каждый вопрос.
- **Типы вопросов:**
  1. **Single Choice:** Выбор одного правильного варианта из предложенных.
  2. **Multiple Choice:** Выбор нескольких правильных вариантов. Балл начисляется только при полностью верном выборе (или настраивается частичный балл).
  3. **Sequence Alignment:** Установление правильной последовательности элементов.

### 3.2. Логика фиксации результатов
- Проверка ответов выполняется автоматически сервером.
- По истечении лимита времени или временного интервала теста прохождение блокируется, а текущие ответы автоматически отправляются на проверку и фиксируются.
- **Интеграция результатов:** В настройках соревнования ADMIN указывает режим учета баллов теста:
  - Включать в общий зачет соревнования (суммировать к баллам за задачи).
  - Учитывать отдельно (только для аналитики или как самостоятельный коэффициент).

## 4. Модуль основного CTF и типы заданий

### 4.1. Разнообразие заданий (Challenges)
Помимо классического ввода текстового флага, система поддерживает следующие типы заданий:
1. **Тестовое задание (Single/Multiple Choice):** Выбор верных ответов (автоматическая проверка).
2. **Последовательность (Sequence):** Расстановка элементов в правильном порядке (автоматическая проверка).
3. **Практическое задание с ручной проверкой (File Upload):** Студент отправляет файл отчета/решения (pdf, zip, txt). ADMIN проверяет его вручную после окончания соревнования и выставляет оценку.

### 4.2. Игровые механики и распределение баллов
- **Режим «First Blood Only» (Первая кровь):** Опциональный флаг для задач. Если включен:
  - Только первый участник, отправивший верное решение (по серверному времени с точностью до миллисекунд), получает баллы за задачу.
  - Все последующие попытки других участников блокируются, а в интерфейсе отображается уведомление, что задача уже решена.
- **Механика первенства при ручной проверке решений:**
  - Скорость загрузки файлов решений влияет на итоговый коэффициент балла:
    - **1-й верный ответ:** получает 100% от выставленного преподавателем балла.
    - **2-й верный ответ:** получает 50% от балла.
    - **3-й и последующие верные ответы:** получают 10% от балла.
  - ADMIN имеет право переопределить этот автоматический процент вручную в панели ручной проверки, указав основание.

## 5. Игровые промокоды

### 5.1. Ограничения и активация
- Промокоды являются глобально уникальными строками (например, `SPRING_PROMO`, `HARDWORK`).
- Промокод является строго одноразовым. После активации одним студентом он помечается как `isUsed` и не может быть активирован повторно кем-либо еще.
- Промокод активируется студентом через личный кабинет.

### 5.2. Типы эффектов промокодов
1. **FIXED_ADD:** Моментальное добавление фиксированного количества баллов к текущему счету студента.
2. **FIXED_SUB:** Списание баллов (штрафной промокод для квестов или балансировки).
3. **DOUBLE_COEFF:** Удвоение набранных баллов на определенный период времени или фиксированное умножение текущего счета.
4. **TEMP_MULTIPLIER:** Применение временного повышающего коэффициента на все задачи, решенные в период действия промокода.

---

## 6. Административный контроль и корректировки

- Преподаватель обладает полным контролем над балансом баллов участников.
- **Ручные корректировки (Score Adjustments):**
  - Возможность начислить бонусные баллы или выписать штраф.
  - Возможность принудительно изменить статус или балл за любое сданное решение.
  - Любое действие обязано сопровождаться указанием основания (причины), которое сохраняется в БД и доступно для просмотра в отчетах.

## 7. Итоговый расчет CTF-показателя и Оценки за семестр

### 7.1. Расчет показателя CTF ($v_2$)
- **Итоговый балл студента за CTF ($N$):** 
  $$N = \sum \text{Баллы за задачи} + \sum \text{Баллы за тесты (если включено)} + \text{Бонусы} - \text{Штрафы} + \text{Промокоды}$$
- **Фактический максимум ($N_{max}$):** Максимальный фактически набранный балл за CTF среди выбранной совокупности студентов (выбирается аналогично лабораторным работам: группа, поток, все).
- **Формула расчета:**
  $$v_2 = \frac{N}{N_{max}}$$

### 7.2. Расчет семестрового коэффициента ($S$) и выставление оценки
- **Итоговый коэффициент ($S$):** Среднее арифметическое двух показателей:
  $$S = \frac{v_1 + v_2}{2}$$
- **Динамическая шкала оценок:**
  - ADMIN настраивает интервалы перевода коэффициента $S$ в оценку через веб-интерфейс. Значения шкал хранятся в БД.
  - Пример шкалы:
    - $[0.00; 0.54] \to 2$ (Неудовлетворительно)
    - $[0.55; 0.69] \to 3$ (Удовлетворительно)
    - $[0.70; 0.84] \to 4$ (Хорошо)
    - $[0.85; 1.00] \to 5$ (Отлично)
  - Платформа автоматически рассчитывает и предлагает рекомендуемую оценку на основе актуальной шкалы.

## 8. Отчеты и Журнал действий (Аудит)

### 8.1. Выгрузка отчетов (Экспорт в CSV/Excel)
Система должна формировать 4 регламентированных отчета:
1. **Итоговый отчет по студентам:** ФИО, Группа, Поток, Баллы за лабораторные ($M$), Коэффициент лаб ($v_1$), Баллы за CTF ($N$), Коэффициент CTF ($v_2$), Итоговый коэффициент ($S$), Рекомендуемая оценка, Сумма бонусов/штрафов.
2. **Отчет по решениям:** Список всех сданных флагов/ответов с фильтрацией по задачам, студентам, корректности и времени сдачи.
3. **Отчет по ручным проверкам:** Список задач с ручной проверкой файлов решений, содержащий ФИО студента, файл, время отправки, полученный балл, примененный коэффициент первенства, комментарий преподавателя.
4. **Отчет по админ-изменениям:** Лог ручных корректировок оценок и баллов преподавателями с указанием авторов изменений и причин.

### 8.2. Сквозной журнал действий (Audit Log)
Все критические действия пользователей логируются в системную таблицу аудита.
- **Записываемые поля:** Дата/время события (серверное), ID пользователя, Роль, Тип действия (Action Type), Идентификатор целевого объекта, Старое значение, Новое значение, IP-адрес запроса.
- **Логируемые события:**
  - Вход в систему (Login), выход (Logout), неуспешные попытки входа.
  - Отправка решений задач и тестов (Submits).
  - Активация промокодов.
  - Изменение статуса студентов (Блокировка, дисквалификация, перевод вне рейтинга).
  - Изменение шкалы оценок.
  - Ручные корректировки баллов.

## 9. Нефункциональные требования

1. **Серверное время как источник истины:** Все проверки времени сдачи, окончания тестов, лимитов First Blood и активаций промокодов используют исключительно системное время сервера.
2. **Реактивное обновление без перезагрузки страниц:** 
  - Таблица лидерборда и статистика сдачи должны обновляться асинхронно с использованием SSE (Server-Sent Events) или WebSockets.
  - Действия одного пользователя (сдача флага, получение First Blood) не должны приводить к обновлению страниц у других пользователей.
  - Ввод ответов студентами в текущие поля форм не должен сбрасываться при фоновом обновлении интерфейса.
3. **Отказоустойчивость при заполнении тестов:** Введенные студентом ответы на вопросы квизов должны периодически сохраняться в LocalStorage или автосохраняться на сервере, предотвращая потерю данных при сбоях связи.
